# LLM检索自定义内存操作（分配/销毁/释放）函数

### NSPA的第一阶段旨在识别目标开源项目中的自定义内存分配和释放函数，识别过程划分为三个步骤：候选函数记录构建、候选函数过滤和LLM语义验证。

### 步骤1：候选函数记录构建

给定目标开源项目，NSPA首先使用Tree-Sitter将源代码解析为具体语法树（Concrete Syntax Tree, CST），从而对函数定义、函数声明以及宏定义等语法结构的准确提取。在解析过程中，NSPA遍历项目中的源文件，并提取每个候选函数实体的基本信息，包括其所在项目、源文件路径、函数签名和直接调用列表。除普通函数外，NSPA还额外提取函数式宏定义，因为C/C++项目中的内存管理操作可能通过宏对底层分配器或释放器进行封装（例如，Vim项目中的\texttt{vim\_malloc}和\texttt{vim\_free}）。若忽略这类函数式宏，可能导致部分项目自定义内存管理接口无法被正确识别。对于每个候选函数实体，NSPA构造一个候选函数记录（Candidate Function Record, CFR）作为后续过滤和验证阶段使用的统一输入结构。它不仅包含传统函数签名信息（即函数名、返回类型和参数列表），还包含函数所属项目、源文件路径、函数体中的直接调用目标列表。直接调用目标列表由Tree-Sitter从函数体中提取，用于记录该候选函数直接调用了哪些函数。与已有工作直接将完整函数体提交给LLM不同\cite{cheng2025boosting,huang2026finding}，NSPA构造的CFR仅保留与内存函数识别相关的轻量级结构信息，能够显著降低后续LLM验证阶段的token消耗。

### 步骤2：候选函数过滤

在构建CFR之后，NSPA对候选函数集合进行保守过滤。该步骤的目标是去除那些明显与内存管理无关的函数，从而减少后续LLM验证的规模。首先，NSPA会删除重复的候选函数记录；当同一位置存在声明和定义时，优先保留包含函数体和直接调用列表的定义记录。随后，NSPA基于多类证据进行加权过滤，而不是仅依赖函数名关键字。第一类是类型证据。例如，返回指针或引用类型、返回对象句柄、包含二级指针参数、输出指针参数或对象指针参数的函数，更可能与内存分配、初始化、释放或销毁有关。第二类是名称证据。NSPA将名称线索划分为强证据和弱证据：\texttt{alloc}、\texttt{malloc}、\texttt{realloc}、\texttt{free}、\texttt{release}、\texttt{destroy}等属于强证据；\texttt{copy}、\texttt{make}、\texttt{clear}、\texttt{close}等属于弱证据，只有当它们同时具备类型、调用或文档证据时才会被保留。这样可以避免把普通拷贝、格式化或输出缓冲刷新函数误认为内存管理接口。

第三类是调用证据。若候选函数直接调用标准内存原语，例如\texttt{malloc}、\texttt{calloc}、\texttt{realloc}、\texttt{strdup}或\texttt{free}，则该函数会被保留为高优先级候选。为了识别项目内部的包装器，NSPA进一步采用固定点调用传播策略：先识别直接调用标准内存原语的核心候选，再将“调用这些核心候选且自身具有局部接口证据”的函数保留下来。传播过程只从较可信的内存候选出发，并且要求接收传播的函数具备名称、类型或文档证据，从而避免把任意调用过分配器的普通业务函数全部扩散进候选集合。第四类是项目API文档和邻近注释。当文档描述中出现“allocate”“free”“release”“destroy”“memory”“heap”“ownership”等语义线索时，该函数会被保留为潜在内存管理函数；相反，若文档明确表明其仅用于日志输出、字符串格式化、比较、哈希、状态查询或统计计数，则降低其过滤得分。实践中，“buffer”一词本身过于宽泛，例如输出缓冲刷新函数并不一定管理堆内存，因此不能单独作为强保留证据，只有与分配、释放、所有权或堆内存语义共同出现时才参与保留判断。

#### 过滤分数计算
当前有两个分数字段：

score：原始证据加权分。主要规则是：

```text
+4.0  直接调用标准分配函数，如 malloc/calloc/realloc/strdup
+4.0  直接调用标准释放函数，如 free/delete
+3.0  强内存名称，如 alloc/free/release/destroy
+1.0  弱内存名称，如 copy/make/clear/close
+1.5  返回指针或引用
+1.5  有二级指针或输出指针参数
+1.0  释放类函数接收对象指针参数
+2.0  注释/文档包含 memory/heap/ownership/free/destroy 等语义
+1.0  函数式宏包装了标准内存原语
+2.5  调用了已识别出的项目内部内存候选函数
-1.5  文档显示偏日志/格式化/状态查询等非内存语义
-1.0  名称像 get/set/log/format 等弱非内存接口
```
新增的 confidence / filter_confidence 是归一化分数：
```text
confidence = clamp(score / 10.0, 0, 1)
```
所以 `score=5.0` 对应 `confidence=0.5`。

需要强调的是，NSPA在过滤阶段采用保守策略，但负证据只用于过滤缺少正证据的候选函数。过于激进的过滤策略容易漏掉真正的自定义分配或释放函数。例如，\texttt{set\_alloc\_policy}虽然包含\texttt{alloc}，但可能只是设置分配策略；而\texttt{buf\_clear}虽然不直接包含\texttt{free}，却可能在内部释放缓冲区。对于这类边界情况，NSPA将其保留到后续LLM语义验证阶段进行进一步区分。

### 步骤3：LLM语义验证

在候选函数过滤之后，NSPA使用LLM对保留下来的候选函数记录进行识别。NSPA仅向LLM提供过滤后的CFR作为判断依据，我们将这种方式称为轻量级记录验证。经过我们的实验验证表明，这些信息足以帮助LLM判断一个函数是否承担内存分配或释放职责。轻量级记录验证也避免了完全依赖函数名猜测的问题。在验证过程中，若一个函数创建新的动态内存对象，并通过返回值或输出参数将其所有权传递给调用者，则该函数应被识别为内存分配函数。若一个函数释放调用者传入的对象、缓冲区或对象内部字段，并终止其生命周期或部分生命周期，则该函数应被识别为内存释放函数。若一个函数销毁整个对象、容器或资源句柄，并在此过程中释放其内部字段，则该函数应被识别为内存销毁函数。相反，若一个函数仅在内部临时申请并释放内存，但没有向调用者返回新内存，也没有释放调用者持有的对象，则不应被标记为自定义内存分配、释放或销毁函数。例如，一个解析函数可能在内部调用\texttt{malloc}申请临时缓冲区，并在返回前调用\texttt{free}释放该缓冲区，但该函数并不承担为调用者分配或释放内存的接口语义，因此NSPA会将其归类为非内存管理函数。相反，若一个名为\texttt{xmalloc}的函数返回指针类型并直接调用\texttt{malloc}，则其更可能是项目自定义分配器；类似地，若\texttt{buffer\_destroy}接收\texttt{Buffer*}类型参数，并直接调用\texttt{free}释放对象内部字段或对象本身，则其更可能是项目自定义释放或销毁函数。为了提高验证结果的可检查性，NSPA要求LLM以JSON格式输出验证结果。输出内容包括函数所在文件路径、函数名、函数类别（内存分配函数、内存释放函数、内存销毁函数或非内存管理函数）、置信度评分以及简短原因。实现中，NSPA按批次提交CFR，并使用断点文件记录每个CFR的验证结果，以便长时间验证任务可中断后继续执行。完成验证后，NSPA将识别出的项目自定义内存函数与标准内存原语共同构建为规则库，用于增强后续指针分析和值流分析。帮助识别新的堆对象创建点、对象生命周期终止点等语义，从而增强堆对象建模、指针传播和跨过程值流传播的准确性。
